## Цель

Понимать декомпозицию на автономные сервисы, trade-offs и типичные ошибки (distributed monolith).

---

## Ключевые концепции

### Определение
- Независимо deployable сервисы
- Каждый владеет своими данными (database per service)
- Общение через network (sync/async)
- Органized around business capabilities

### Декомпозиция
- По bounded context (DDD)
- По subdomain
- **Не** по техническим слоям (UI service, DB service — антипаттерн)

### Коммуникация
| Sync | Async |
|------|-------|
| REST, gRPC | Events, message queues |
| Простота | Loose coupling, resilience |
| Coupling, cascading failures | Eventual consistency |

### Cross-cutting concerns
- **Service discovery** — Consul, K8s DNS
- **API Gateway** — routing, auth, rate limit
- **Config management** — externalized config
- **Distributed tracing** — correlation ID

### Anti-patterns
- Distributed monolith
- Shared database
- Chatty services
- No API versioning
