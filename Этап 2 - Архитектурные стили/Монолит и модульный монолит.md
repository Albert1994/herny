# Монолит и модульный монолит

---

## Цель

Понимать, когда монолит — правильный выбор, и как строить модульный монолит как stepping stone к распределённой системе.

---

## Ключевые концепции

### Классический монолит
- Один deployable unit
- Shared database (часто)
- Простота: один repo, один pipeline, ACID transactions

### Modular Monolith
- Один deploy, но **чёткие module boundaries**
- Модули общаются через public API (не direct DB access)
- Каждый модуль ≈ bounded context
- Может эволюционировать в microservices (extract module → service)

### Когда монолит лучше
- Команда < 10–15 человек
- Продукт на стадии PMF / MVP
- Сильная связность данных (нужны ACID транзакции)
- Нет ops-экспертизы для distributed systems

### Modular Monolith patterns
- Package-by-feature (не package-by-layer)
- Module-local schema (logical separation in one DB)
- ArchUnit / dependency rules in CI

## Практика

1. Разбить монолит на 3 модуля с enforced boundaries
2. CI check: module A не импортирует internal module B
3. Документ: «почему не microservices (пока)»
