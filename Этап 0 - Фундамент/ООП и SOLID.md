## Цель

Уверенно применять принципы объектно-ориентированного проектирования как основу для модульной и расширяемой архитектуры.

---

## Ключевые концепции

### ООП
- **Инкапсуляция** — скрытие внутреннего состояния, публичный контракт
- **Абстракция** — выделение существенного, игнорирование деталей
- **Наследование vs композиция** — предпочитай композицию
- **Полиморфизм** — подстановка реализаций через интерфейсы

### SOLID
| Принцип | Суть |
|---------|------|
| **S** — Single Responsibility | Один класс/модуль — одна причина для изменения |
| **O** — Open/Closed | Открыт для расширения, закрыт для модификации |
| **L** — Liskov Substitution | Подтипы заменяют базовые типы без поломки контракта |
| **I** — Interface Segregation | Много узких интерфейсов лучше одного «толстого» |
| **D** — Dependency Inversion | Зависимость от абстракций, не от реализаций |

## Практика

1. Взять существующий класс с 3+ ответственностями → разбить по SRP
2. Добавить новую фичу через OCP (strategy/decorator), не меняя существующий код
3. Найти нарушение LSP в open-source проекте и описать последствия
