# JOIN и N+1

**Раздел:** [[0. Реляционные БД|Реляционные БД (SQL)]]

---

## Главная мысль

JOIN — сильная сторона SQL, но в коде следи за количеством round-trip'ов к БД. Один запрос с JOIN почти всегда лучше N отдельных.

---

## Типы JOIN

Объединение таблиц по связи (обычно FK).

| Тип | Что возвращает |
|-----|----------------|
| **INNER JOIN** | Только строки с совпадением в обеих таблицах |
| **LEFT JOIN** | Все из левой + совпадения справа (NULL, если нет) |
| **RIGHT JOIN** | Все из правой + совпадения слева |
| **FULL OUTER JOIN** | Все из обеих, NULL где нет пары |
| **CROSS JOIN** | Декартово произведение (каждая с каждой) |

---

## N+1 problem

Антипаттерн ORM: 1 запрос за список сущностей + N запросов за связанные данные для каждой строки.

```sql
-- Плохо (N+1): 1 запрос orders + 100 запросов users
SELECT * FROM orders;           -- 100 заказов
SELECT * FROM users WHERE id=1; -- × 100 раз

-- Хорошо: 1 запрос с JOIN
SELECT o.*, u.name FROM orders o JOIN users u ON o.user_id = u.id;
```

---

## Решения N+1

- `JOIN` в SQL-запросе
- **Eager loading** в ORM (`include`, `preload`)
- **DataLoader** — батчинг запросов в GraphQL
- **Денормализация** для read-моделей (см. [[1. Нормализация и денормализация]])
