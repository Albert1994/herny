# Red-Black и AVL Tree

**Раздел:** [[../0. Деревья|Деревья]]

---

## Коротко

**AVL Tree** и **Red-Black Tree** — сбалансированные бинарные деревья поиска.

Бинарное значит: у узла максимум два ребёнка.

```text
        50
       /  \
     30    70
    / \    / \
  20  40  60  80
```

Они гарантируют, что дерево не выродится в список.

---

## Для чего нужно

Эти деревья нужны для упорядоченного множества или map в памяти:

- быстро найти ключ;
- быстро вставить ключ;
- получить следующий/предыдущий ключ;
- пройти значения в отсортированном порядке.

---

## Где используется

- `std::map` / `std::set` в C++ часто реализованы как Red-Black Tree.
- Linux kernel использует red-black trees для некоторых структур.
- Java `TreeMap` — Red-Black Tree.
- In-memory индексы и служебные структуры storage engines.
- MemTable в некоторых LSM-подобных системах может быть похожей структурой, хотя часто выбирают Skip List.

---

## AVL против Red-Black

| Вопрос | AVL Tree | Red-Black Tree |
|---|---|---|
| Баланс | строже | слабее |
| Поиск | немного быстрее | немного хуже AVL |
| Вставка/удаление | больше вращений | меньше вращений |
| Где удобно | read-heavy in-memory | general-purpose map/set |

---

## Как работает получение

Ищем `60`:

```text
        50
       /  \
     30    70
          /  \
        60    80
```

Шаги:

1. `60 > 50` → идём вправо.
2. `60 < 70` → идём влево.
3. Нашли `60`.

Сложность: `O(log n)`.

---

## Как работает вставка

Вставляем `65`.

1. Идём как в обычном binary search tree:

```text
50 -> 70 -> 60 -> right
```

2. Добавляем новый узел:

```text
        50
       /  \
     30    70
          /  \
        60    80
          \
          65
```

3. Проверяем баланс.
4. Если дерево стало слишком перекошенным, делаем **rotation**.

Rotation — локальная перестановка узлов, которая уменьшает высоту ветки.

---

## Почему не используют как основной дисковый индекс

У бинарного дерева маленькая «ширина» узла: максимум 2 ребёнка.

На миллионах строк высота будет намного больше, чем у B+ Tree.

```text
Red-Black / AVL: много маленьких переходов
B+ Tree: мало переходов, но каждый узел широкий
```

Для диска это критично: каждый переход может означать чтение страницы.

Поэтому для дисковых индексов обычно выбирают B+ Tree.

---

## Практический пример

В приложении можно использовать дерево для leaderboard-like структуры:

```text
score -> player
100 -> Alice
120 -> Bob
150 -> Carol
```

Операции:

- вставить новый score;
- найти всех игроков с `score >= 120`;
- получить top-N.

В Redis Sorted Set похожая задача решается через Skip List, а не Red-Black Tree, но идея упорядоченной in-memory структуры та же.

---

## Как объяснить на интервью

AVL и Red-Black — сбалансированные бинарные деревья поиска для in-memory структур. AVL держит баланс строже и быстрее ищет, Red-Black делает меньше перестроений при записи. Для дисковых индексов они хуже B+ Tree, потому что имеют большую высоту и больше random I/O.
