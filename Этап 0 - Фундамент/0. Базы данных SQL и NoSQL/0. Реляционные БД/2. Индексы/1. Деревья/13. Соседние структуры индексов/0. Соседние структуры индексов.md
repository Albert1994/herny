## Коротко

Не все индексы — деревья.

Рядом с деревьями важно знать несколько структур:

- Hash index;
- GIN / inverted index;
- BRIN;
- Bitmap index.

Они не являются «видами деревьев», но часто сравниваются с B+ Tree, LSM и GiST, потому что решают похожую задачу: ускоряют доступ к данным.

---

## Hash index

### Для чего нужен

Hash index нужен для быстрого поиска по равенству:

```sql
WHERE email = 'alice@example.com'
```

### Как выглядит

```text
hash(email) -> bucket -> row pointers
```

### Как работает вставка

1. Считаем hash от ключа.
2. Находим bucket.
3. Кладём туда указатель на строку.
4. Если bucket переполнен, используется overflow или resize, зависит от реализации.

### Как работает получение

1. Считаем hash от ключа.
2. Идём в bucket.
3. Проверяем элементы bucket, потому что возможны коллизии.

### Ограничение

Hash index не помогает для:

```sql
WHERE created_at BETWEEN ...
ORDER BY created_at
```

Потому что hash разрушает порядок.

## GIN / Inverted Index

### Для чего нужен

GIN нужен, когда одна строка содержит много элементов, и нужно искать по внутренним элементам:

- full-text search;
- JSONB;
- arrays;
- tags.

### Как выглядит

```text
"database" -> [doc1, doc5, doc99]
"index"    -> [doc1, doc3, doc99]
```

Это называется inverted index: не документ → слова, а слово → документы.

### Как работает вставка

Документ:

```text
"database index tutorial"
```

1. Разбиваем на лексемы: `database`, `index`, `tutorial`.
2. Для каждой лексемы добавляем `doc_id` в posting list.
3. Posting lists могут храниться сжато и индексироваться внутренними структурами.

### Как работает получение

Запрос:

```sql
WHERE to_tsvector(body) @@ to_tsquery('database & index')
```

1. Находим posting list для `database`.
2. Находим posting list для `index`.
3. Пересекаем списки.
4. Получаем документы, где есть оба термина.

## BRIN

### Для чего нужен

BRIN — Block Range Index. Он хранит сводку по диапазонам блоков таблицы.

Хорош для огромных таблиц, где данные физически идут по порядку:

- логи;
- события;
- time-series;
- append-only таблицы.

### Как выглядит

```text
block range 1: created_at min=10:00 max=10:05
block range 2: created_at min=10:05 max=10:10
block range 3: created_at min=10:10 max=10:15
```

### Как работает вставка

1. Новые строки добавляются в конец таблицы.
2. BRIN обновляет min/max для диапазона блоков.
3. Индекс остаётся очень маленьким.

### Как работает получение

Запрос:

```sql
WHERE created_at BETWEEN '10:06' AND '10:08'
```

BRIN смотрит на summary:

- block range 1: не подходит;
- block range 2: может подходить;
- block range 3: не подходит.

Затем база читает только потенциально нужные блоки.

## Bitmap index

### Для чего нужен

Bitmap index хорошо работает для колонок с низкой кардинальностью:

- `status`;
- `gender`;
- `is_active`;
- `country`, если значений мало относительно строк.

### Как выглядит

```text
status = 'new'      -> 1 0 0 1 0
status = 'paid'     -> 0 1 0 0 1
status = 'shipped'  -> 0 0 1 0 0
```

Каждый бит соответствует строке.

### Как работает получение

Запрос:

```sql
WHERE status = 'paid' AND is_active = true
```

Система берёт две битовые карты и делает побитовое `AND`.

Это очень быстро в OLAP.

### Почему плохо для частых UPDATE

Если строка часто меняет статус, нужно менять битовые карты. В OLTP это может быть дорого и конфликтно.

---

## Как выбирать между ними и деревьями

| Задача | Лучше смотреть на |
|---|---|
| Равенство и диапазоны | B+ Tree |
| Только равенство | Hash |
| Full-text / JSONB / arrays | GIN |
| Огромная append-only таблица | BRIN |
| OLAP и низкая кардинальность | Bitmap |
| Геоданные / ranges | GiST/R-Tree |
| Write-heavy storage | LSM Tree |
