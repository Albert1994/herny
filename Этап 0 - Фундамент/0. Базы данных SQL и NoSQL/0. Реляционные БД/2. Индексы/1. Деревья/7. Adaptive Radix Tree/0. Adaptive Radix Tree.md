# Adaptive Radix Tree

**Раздел:** [[../0. Деревья|Деревья]]  
**Контекст:** [[../../0. Индексы|Индексы]]

---

## Коротко

**Adaptive Radix Tree (ART)** — оптимизированный radix tree для in-memory баз данных.

Он хранит ключи как последовательность байтов, но адаптирует размер узла под количество детей.

Идея: не держать огромный массив на 256 детей в каждом узле, если реально детей всего 2–3.

---

## Для чего нужно

ART нужен для быстрых in-memory индексов, особенно когда:

- ключи строковые или байтовые;
- нужны prefix/range операции;
- важна cache locality;
- B+ Tree в памяти не всегда оптимален;
- обычный trie слишком прожорлив по памяти.

---

## Где используется

- HyPer и in-memory database research.
- Некоторые аналитические и embedded storage engines.
- Системы, где индекс полностью или преимущественно находится в RAM.

---

## Как выглядит

ART использует разные типы узлов:

```text
Node4    - до 4 детей
Node16   - до 16 детей
Node48   - до 48 детей
Node256  - до 256 детей
```

Если у узла мало детей, он компактный. Если детей становится больше, узел расширяется.

```text
root
 └── 'u'
      └── 's'
           └── 'e'
                └── 'r'
                     ├── ':' -> ...
```

---

## Как работает вставка

Вставляем ключ:

```text
user:1001
```

Шаги:

1. Представляем ключ как массив байтов.
2. Идём по дереву байт за байтом.
3. Если нужного перехода нет — создаём ребёнка.
4. Если узел переполнен, например Node4 получает пятого ребёнка, он превращается в Node16.
5. В листе сохраняем значение или указатель на строку.

Пример адаптации:

```text
Node4:  дети a,b,c,d
Вставляем e
Node4 -> Node16
```

---

## Как работает получение

Ищем ключ:

```text
user:1001
```

1. Берём первый байт `u`.
2. В текущем узле ищем ребёнка `u`.
3. Переходим к следующему байту.
4. Повторяем до конца ключа.
5. Если дошли до листа и ключ совпал — значение найдено.

Сложность похожа на trie: `O(k)`, где `k` — длина ключа. Но ART оптимизирован под CPU cache и память.

---

## Path compression

Если есть длинная цепочка без ветвления, ART может хранить её как общий префикс.

Вместо:

```text
u -> s -> e -> r -> :
```

можно хранить:

```text
"user:" -> дальше ветвление
```

Это уменьшает глубину дерева.

---

## Практический пример

Представь in-memory таблицу с ключами:

```text
customer:1001
customer:1002
customer:1003
order:5001
order:5002
```

ART хорошо хранит общие префиксы:

```text
root
├── "customer:" -> 1001,1002,1003
└── "order:"    -> 5001,5002
```

Для prefix scan `customer:` дерево быстро переходит к нужному поддереву.

---

## ART против B+ Tree

| Вопрос | ART | B+ Tree |
|---|---|---|
| Среда | RAM | диск и RAM |
| Ключи | байты/строки/префиксы | упорядоченные значения |
| Сложность | `O(k)` | `O(log n)` |
| Сильная сторона | cache-friendly prefix lookup | универсальность и диапазоны |
| Где чаще | in-memory DB | классические RDBMS |

---

## Как объяснить на интервью

ART — это сжатый radix tree, оптимизированный под память и CPU cache. Он адаптирует размер узлов к фактическому числу детей: Node4, Node16, Node48, Node256. Благодаря этому он экономит память по сравнению с обычным trie и хорошо подходит для in-memory индексов.
