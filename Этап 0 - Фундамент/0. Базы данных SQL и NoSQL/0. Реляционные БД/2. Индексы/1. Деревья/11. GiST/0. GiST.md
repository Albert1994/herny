## Коротко

**GiST** расшифровывается как **Generalized Search Tree**.

Это не одно конкретное дерево, а фреймворк для построения деревьев поиска в PostgreSQL.

GiST говорит: «если ты умеешь описать, как сравнивать, объединять и отсекать значения, PostgreSQL сможет построить для них индекс».

---

## Для чего нужно

Обычный B+ Tree знает, как сравнивать значения через `<`, `=`, `>`.

Но как индексировать:

- геометрию;
- диапазоны дат;
- пересечение интервалов;
- nearest neighbor;
- полнотекстовые похожести;
- кастомные типы?

Для этого нужен более общий механизм. GiST предоставляет такой механизм.

---

## Где используется

В PostgreSQL GiST применяется для:

- PostGIS geometry/geography;
- range types: `tsrange`, `int4range`, `daterange`;
- nearest-neighbor search;
- exclusion constraints;
- некоторых расширений.

---

## Как выглядит

GiST-индекс похож на дерево, где каждый внутренний узел хранит «обобщённое описание» дочерних значений.

Для геометрии это может быть bounding box:

```text
                 bbox all
               /          \
          bbox A          bbox B
          /   \           /   \
       obj1  obj2      obj3  obj4
```

Для range type это может быть диапазон, покрывающий дочерние диапазоны:

```text
[2026-01-01, 2026-12-31]
   /                      \
[Jan-Mar]              [Apr-Dec]
```

---

## Как работает получение

Запрос:

```sql
SELECT * FROM bookings
WHERE period && tsrange('2026-06-01', '2026-06-10');
```

Оператор `&&` значит «диапазоны пересекаются».

Шаги:

1. GiST смотрит на обобщённое значение в корне.
2. Функция `consistent` отвечает: может ли в этой ветке быть совпадение?
3. Если нет — ветка отбрасывается.
4. Если да — спускаемся ниже.
5. На листьях проверяем реальные значения.

---

## Как работает вставка

Вставляем новый диапазон:

```text
[2026-06-01, 2026-06-10]
```

GiST должен выбрать, в какую ветку его положить.

Для этого используется идея `penalty`:

- насколько придётся расширить текущий узел, если добавить туда новое значение;
- выбирается ветка с минимальной «ценой» расширения.

После вставки узлы на пути вверх обновляют своё обобщённое значение через `union`.

Если узел переполнен, GiST использует `picksplit`, чтобы разделить значения между двумя узлами.

---

## Практический пример range index

```sql
CREATE TABLE bookings (
  id bigserial PRIMARY KEY,
  room_id bigint NOT NULL,
  period tsrange NOT NULL
);

CREATE INDEX idx_bookings_period
ON bookings USING GIST (period);
```

Запрос:

```sql
SELECT * FROM bookings
WHERE period && tsrange('2026-06-01', '2026-06-10');
```

GiST помогает не проверять все бронирования.

---

## Exclusion constraint

GiST часто используют для запрета пересекающихся бронирований:

```sql
CREATE EXTENSION IF NOT EXISTS btree_gist;

ALTER TABLE bookings
ADD CONSTRAINT no_overlap
EXCLUDE USING GIST (
  room_id WITH =,
  period WITH &&
);
```

Смысл: для одной комнаты нельзя иметь пересекающиеся периоды.

---

## Как объяснить на интервью

GiST — это обобщённый фреймворк деревьев поиска в PostgreSQL. Он позволяет строить индексы для типов, где обычного линейного порядка недостаточно: геометрия, ranges, nearest neighbor. Логика задаётся через функции вроде `consistent`, `union`, `penalty`, `picksplit`.
