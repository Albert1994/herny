## Главная мысль

Индекс в базе данных — это не «ускоритель вообще», а структура данных под конкретный способ доступа к данным.

Деревья нужны, чтобы быстро отвечать на вопросы:

- где лежит строка с нужным ключом;
- какие строки попадают в диапазон;
- как обойти значения в отсортированном порядке;
- как найти объекты в пространстве;
- как проверить целостность данных между репликами;
- как выдержать большой поток записей.

Если очень грубо:

- **B+ Tree** — универсальный индекс для OLTP: `=`, `BETWEEN`, `ORDER BY`.
- **LSM Tree** — когда записей очень много и нужно писать быстро.
- **R-Tree / Quadtree** — когда данные пространственные: точки, полигоны, карты.
- **Trie / Radix / ART** — когда важен префикс: строки, IP, autocomplete.
- **Merkle Tree** — когда нужно сравнивать не значения, а их хеши: репликация, Git, blockchain.

---

## Базовая идея дерева поиска

Дерево делит пространство поиска на части.

Вместо того чтобы проверять все строки:

```text
1, 2, 3, 4, 5, 6, 7, 8, 9, ...
```

мы каждый раз выбираем нужную ветку:

```text
             [50]
            /    \
        < 50      > 50
```

В БД это особенно важно из-за диска: чтение одной страницы с диска намного дороже, чем сравнение чисел в памяти. Поэтому СУБД любят деревья с **широкими узлами**: в одном узле лежит много ключей, а высота дерева остаётся маленькой.

## Как выбирать структуру

| Если запрос выглядит так | Обычно подходит | Почему |
|---|---|---|
| `WHERE id = 42` | B+ Tree или Hash | точечный поиск |
| `WHERE created_at BETWEEN ...` | B+ Tree или BRIN | диапазон и порядок |
| `ORDER BY created_at LIMIT 100` | B+ Tree | индекс уже отсортирован |
| `INSERT 100000 events/sec` | LSM Tree | запись идёт последовательно и буферизуется |
| `WHERE location within polygon` | R-Tree / GiST | отсечение областей по bounding boxes |
| `WHERE prefix LIKE 'abc%'` | Trie / Radix / ART | поиск по символам префикса |
| `repair replicas` | Merkle Tree | быстро найти отличающиеся части данных |
| `status IN (...)` в OLAP | Bitmap | битовые операции по колонкам низкой кардинальности |

## Важное различие: диск против памяти

### На диске

На диске дорого прыгать по маленьким узлам. Поэтому классические бинарные деревья (`AVL`, `Red-Black`) плохо подходят для дисковых индексов: у них много уровней.

Для диска лучше:

- B+ Tree;
- LSM Tree;
- R-Tree/GiST;
- BRIN для очень больших физически упорядоченных таблиц.

### В памяти

В RAM случайный доступ намного дешевле, поэтому можно использовать:

- Red-Black Tree;
- AVL Tree;
- Skip List;
- ART;
- T-Tree.

## Мини-карта обучения

1. Сначала изучи [[1. B-Tree и B+ Tree/0. B-Tree и B+ Tree|B-Tree и B+ Tree]] — это база для понимания обычных индексов.
2. Затем [[3. LSM Tree/0. LSM Tree|LSM Tree]] — чтобы понять Cassandra/RocksDB/LevelDB.
3. Потом [[4. R-Tree и R-star Tree/0. R-Tree и R-star Tree|R-Tree]] и [[11. GiST/0. GiST|GiST]] — для spatial/range индексов.
4. После этого [[6. Trie Radix Patricia/0. Trie Radix Patricia|Trie/Radix]] и [[7. Adaptive Radix Tree/0. Adaptive Radix Tree|ART]] — для строк и prefix lookup.
5. В конце [[10. Merkle Tree/0. Merkle Tree|Merkle Tree]] — для распределённых систем и проверки целостности.
6. Для spatial/vector задач изучи [[14. K-d Tree/0. K-d Tree|K-d Tree]] и [[17. HNSW Graph Index/0. HNSW Graph Index|HNSW Graph Index]].
7. Для интервалов и подстрок изучи [[15. Interval и Segment Tree/0. Interval и Segment Tree|Interval/Segment Tree]] и [[16. Suffix Tree и Suffix Array/0. Suffix Tree и Suffix Array|Suffix Tree/Array]].
