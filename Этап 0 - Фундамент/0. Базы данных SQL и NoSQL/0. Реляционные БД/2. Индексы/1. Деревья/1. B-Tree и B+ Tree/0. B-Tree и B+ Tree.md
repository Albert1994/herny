# B-Tree и B+ Tree

**Раздел:** [[../0. Деревья|Деревья]]  
**Контекст:** [[../../0. Индексы|Индексы]]

---

## Коротко

**B-Tree** — сбалансированное дерево поиска, где один узел хранит не один ключ, а много ключей.

**B+ Tree** — вариант B-Tree, который чаще всего используют СУБД:

- внутренние узлы хранят ключи для навигации;
- реальные данные или указатели на строки лежат в листьях;
- листья связаны между собой списком, поэтому диапазоны читаются быстро.

Если ты видишь обычный индекс в PostgreSQL, MySQL или SQLite, в большинстве случаев думай: «под капотом что-то очень похожее на B+ Tree».

---

## Для чего нужно

B+ Tree решает главную проблему таблицы: без индекса СУБД вынуждена читать строки подряд.

Например:

```sql
SELECT * FROM users WHERE email = 'alice@example.com';
```

Без индекса база может пройти всю таблицу. С B+ Tree она делает несколько переходов по дереву и попадает в нужный лист.

B+ Tree особенно полезен для:

- поиска по равенству: `id = 10`;
- диапазонов: `created_at BETWEEN ...`;
- сортировки: `ORDER BY created_at`;
- `MIN` / `MAX`;
- composite index: `(user_id, created_at)`.

---

## Где используется

- PostgreSQL: стандартный `btree` индекс.
- MySQL InnoDB: clustered index по primary key реализован как B+ Tree.
- SQLite: таблицы и индексы используют B-tree/B+tree-подобные структуры.
- Файловые системы и key-value storage engines.

---

## Как выглядит

Представим индекс по `id`.

```text
                         [30 | 60]
                       /     |      \
              [10 | 20]  [40 | 50]  [70 | 80 | 90]
                 |           |            |
              row ptrs    row ptrs     row ptrs
```

Корень говорит: «если ключ меньше 30 — иди влево; если между 30 и 60 — в середину; если больше 60 — вправо».

В B+ Tree листья связаны:

```text
[10|20] <-> [40|50] <-> [70|80|90]
```

Это важно для диапазонов. Найдя начало диапазона, база просто идёт по листьям вправо.

---

## Как работает получение по ключу

Запрос:

```sql
SELECT * FROM users WHERE id = 45;
```

Дерево:

```text
                         [30 | 60]
                       /     |      \
              [10 | 20]  [40 | 50]  [70 | 80 | 90]
```

Шаги:

1. Смотрим корень `[30 | 60]`.
2. `45` больше `30`, но меньше `60` → идём в среднюю ветку.
3. Попадаем в лист `[40 | 50]`.
4. В листе находим позицию, где должен быть `45`.
5. Если ключ найден — берём указатель на строку или саму строку.

Сложность: `O(log n)`, но важнее другое: высота B+ Tree обычно маленькая. Даже на миллионах и миллиардах строк часто нужно всего несколько чтений страниц.

---

## Как работает получение диапазона

Запрос:

```sql
SELECT * FROM orders
WHERE created_at BETWEEN '2026-01-01' AND '2026-01-31'
ORDER BY created_at;
```

Шаги:

1. Дерево ищет первый лист, где лежит `2026-01-01`.
2. Затем идёт по связанным листьям вправо.
3. Останавливается после `2026-01-31`.

База не сортирует результат отдельно, если порядок индекса уже подходит.

---

## Как работает вставка

Допустим, лист может хранить максимум 3 ключа.

Было:

```text
[10 | 20 | 30]
```

Вставляем `25`.

1. Находим лист, куда должен попасть ключ.
2. Вставляем в отсортированное место:

```text
[10 | 20 | 25 | 30]
```

3. Лист переполнен → делим его на два:

```text
[10 | 20]    [25 | 30]
```

4. Ключ-разделитель `25` поднимается в родительский узел.
5. Если родитель тоже переполнен, деление может подняться выше, вплоть до создания нового корня.

Главная идея: дерево всегда остаётся сбалансированным — все листья примерно на одной глубине.

## Практический пример

```sql
CREATE TABLE users (
  id bigint PRIMARY KEY,
  email text NOT NULL,
  created_at timestamp NOT NULL
);

CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_users_created_at ON users(created_at);
```

Теперь:

```sql
SELECT * FROM users WHERE email = 'alice@example.com';
```

может использовать `idx_users_email`.

А запрос:

```sql
SELECT * FROM users
WHERE created_at >= now() - interval '7 days'
ORDER BY created_at;
```

может использовать `idx_users_created_at`, потому что B+ Tree хранит ключи в порядке.

## Важные ограничения

### 1. Левый префикс composite index

Индекс:

```sql
CREATE INDEX idx_orders_user_date ON orders(user_id, created_at);
```

Хорошо работает для:

```sql
WHERE user_id = 10
WHERE user_id = 10 AND created_at > now() - interval '7 days'
```

Но хуже для:

```sql
WHERE created_at > now() - interval '7 days'
```

Потому что первый ключ в дереве — `user_id`.

### 2. Много случайных вставок

Если primary key случайный (`uuid v4`), вставки разлетаются по разным листьям. Это может приводить к page split и фрагментации.

### 3. Индекс ускоряет чтение, но замедляет запись

При `INSERT`, `UPDATE`, `DELETE` база должна обновить не только таблицу, но и каждый связанный индекс.

---

## Как объяснить на интервью

B+ Tree — это широкое сбалансированное дерево, оптимизированное под страницы диска. Внутренние узлы направляют поиск, данные лежат в листьях, листья связаны между собой. Поэтому оно хорошо работает и для точечного поиска, и для диапазонов, и для сортировки.
