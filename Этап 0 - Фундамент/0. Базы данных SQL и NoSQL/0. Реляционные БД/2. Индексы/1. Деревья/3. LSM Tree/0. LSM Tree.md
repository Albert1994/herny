# LSM Tree

**Раздел:** [[../0. Деревья|Деревья]]

---

## Коротко

**LSM Tree** расшифровывается как **Log-Structured Merge Tree**.

Это структура для систем, где очень много записей. Она делает запись дешёвой: сначала пишет в память и журнал, потом сбрасывает данные на диск большими отсортированными файлами.

Важно: LSM Tree — это не одно дерево в привычном виде. Это набор уровней:

```text
MemTable -> SSTable L0 -> SSTable L1 -> SSTable L2 -> ...
```

---

## Для чего нужно

B+ Tree при каждой вставке должен найти нужный лист и изменить страницу. Если записи идут случайно, это много random I/O.

LSM Tree решает проблему иначе:

1. Запись сначала идёт в память.
2. На диск данные пишутся последовательно.
3. Старые файлы потом сливаются фоновым процессом.

Это подходит для write-heavy систем:

- события;
- метрики;
- логи;
- IoT;
- time-series;
- большие key-value хранилища.

---

## Где используется

- Cassandra.
- ScyllaDB.
- RocksDB.
- LevelDB.
- Bigtable-подобные системы.
- Многие embedded key-value engines.

---

## Как выглядит

```text
                 запись
                   |
                   v
        +---------------------+
        | MemTable в памяти   |
        +---------------------+
                   |
              flush на диск
                   v
        +---------------------+
        | SSTable L0          |
        +---------------------+
                   |
              compaction
                   v
        +---------------------+
        | SSTable L1, L2 ...  |
        +---------------------+
```

**SSTable** — Sorted String Table: неизменяемый отсортированный файл на диске.

---

## Как работает вставка

Допустим, мы пишем событие:

```sql
INSERT INTO events(device_id, ts, payload)
VALUES ('sensor-1', now(), '{"temp": 21}');
```

Внутри LSM-подобной системы обычно происходит так:

1. Запись попадает в **WAL** — write-ahead log, чтобы не потерять данные при падении.
2. Запись добавляется в **MemTable** — структуру в памяти, часто skip list или red-black tree.
3. Когда MemTable становится большой, она замораживается.
4. Замороженная MemTable сбрасывается на диск как отсортированный SSTable.
5. На диске появляются несколько SSTable.
6. Фоновый процесс **compaction** сливает SSTable, удаляет старые версии и tombstone-записи.

Главная выгода: диск получает большие последовательные записи, а не тысячи случайных мелких изменений.

---

## Как работает получение

Запрос:

```text
GET sensor-1:2026-06-17T10:00:00
```

Шаги:

1. Сначала смотрим MemTable.
2. Если не нашли — смотрим immutable MemTable, если она есть.
3. Потом проверяем SSTable на уровне L0.
4. Затем L1, L2 и дальше.
5. Чтобы не открывать каждый файл, используются **Bloom filters**.

Bloom filter быстро отвечает: «ключа точно нет» или «ключ, возможно, есть».

---

## Пример на практике

Представь сервис метрик:

```text
100 000 записей в секунду:
cpu.host1.2026-06-17T10:00:00 = 72%
cpu.host2.2026-06-17T10:00:00 = 55%
...
```

Если каждую запись пытаться вставлять прямо в B+ Tree на диске, будет много random write.

LSM делает иначе:

```text
новые метрики -> память -> пачка на диск -> фоновое слияние
```

Поэтому Cassandra/RocksDB хорошо подходят для большого потока записей.

---

## Compaction — сердце LSM

Со временем на диске появляются файлы с разными версиями одного ключа:

```text
L0: user:1 = Alice v3
L1: user:1 = Alice v2
L2: user:1 = Alice v1
```

Compaction оставляет актуальную версию:

```text
user:1 = Alice v3
```

Удаление работает через **tombstone** — специальную запись «ключ удалён». Физически старые значения исчезают позже, во время compaction.

---

## Trade-offs

| Плюс | Цена |
|---|---|
| Очень быстрая запись | Чтение может проверять несколько уровней |
| Последовательный disk write | Нужен compaction |
| Хорошо для больших потоков данных | Возможны write amplification и read amplification |
| Удобно для immutable файлов | Удаления не мгновенно освобождают место |

---

## Как объяснить на интервью

LSM Tree оптимизирует запись. Новые данные сначала пишутся в память и WAL, затем сбрасываются на диск как отсортированные immutable файлы. Чтение проверяет память и уровни SSTable, а compaction фоново сливает файлы и удаляет старые версии. Это хорошо для write-heavy систем, но усложняет чтение и требует настройки compaction.
