## Коротко

**Merkle Tree** — хеш-дерево.

Листья хранят хеши данных, а каждый внутренний узел хранит хеш от своих детей.

```text
             root_hash
             /       \
       hash(A,B)   hash(C,D)
        /    \       /    \
     h(A)   h(B)  h(C)   h(D)
```

Если меняется один лист, меняются хеши по пути до корня.

---

## Для чего нужно

Merkle Tree нужен не для обычного поиска `WHERE id = ...`, а для проверки целостности и сравнения больших наборов данных.

Он отвечает на вопросы:

- одинаковые ли данные на двух репликах;
- какой блок данных отличается;
- входит ли транзакция в блок;
- не повреждён ли объект;
- можно ли проверить часть данных без скачивания всего набора.

---

## Где используется

- Git: объекты связаны через хеши.
- Blockchain: проверка транзакций в блоке.
- Cassandra/Dynamo-подобные системы: anti-entropy repair.
- Распределённые хранилища.
- Content-addressed storage.

---

## Как выглядит на практике

Есть 4 блока данных:

```text
A = "user:1"
B = "user:2"
C = "user:3"
D = "user:4"
```

Считаем хеши:

```text
hA = hash(A)
hB = hash(B)
hC = hash(C)
hD = hash(D)
```

Строим дерево:

```text
root = hash(hash(hA + hB) + hash(hC + hD))
```

Если `C` изменится, изменятся:

```text
hC -> hash(C,D) -> root
```

---

## Как работает вставка или изменение

Добавляем новый блок `E`.

1. Считаем `hE = hash(E)`.
2. Добавляем его как новый лист.
3. Пересчитываем родительские хеши.
4. Пересчитываем root hash.

Если дерево используется для immutable данных, часто не «изменяют» старое дерево, а создают новую версию дерева с новыми хешами.

---

## Как работает получение/проверка

### Проверка, что элемент входит в набор

Чтобы доказать, что `C` входит в дерево, не нужно отдавать все данные.

Нужно дать:

- сам `C` или `hC`;
- соседний хеш `hD`;
- хеш соседней ветки `hash(A,B)`;
- root hash.

Проверяющий сам пересчитает путь до root.

Если root совпал — доказательство валидно.

---

## Как сравнивать реплики

Допустим, есть две реплики Cassandra.

Реплика 1:

```text
root = abc123
```

Реплика 2:

```text
root = abc123
```

Если root одинаковый, вероятно данные одинаковые.

Если root разный:

1. Сравниваем хеши детей.
2. Находим ветку, где хеш отличается.
3. Спускаемся ниже.
4. Доходим до конкретного диапазона ключей, который нужно синхронизировать.

Это намного дешевле, чем сравнивать все строки.

## Практический пример Git

В Git commit указывает на tree, tree указывает на blobs и другие trees. Всё адресуется хешами.

Если файл изменился:

```text
blob hash меняется
parent tree hash меняется
commit hash меняется
```

Поэтому Git может быстро обнаружить изменение содержимого.

---

## Важные ограничения

- Merkle Tree не ускоряет обычный SQL-поиск.
- Нужно выбрать способ разбиения данных на листья.
- Хеш-функция должна быть устойчивой к коллизиям для security-sensitive задач.
- Для динамических данных нужно продумать обновление дерева.

---

## Как объяснить на интервью

Merkle Tree — это дерево хешей. Листья — хеши данных, внутренние узлы — хеши от детей. Корень является компактным отпечатком всего набора. Если два корня различаются, можно спускаться по дереву и быстро найти отличающуюся часть данных. Поэтому Merkle Tree используют в Git, blockchain и anti-entropy repair в распределённых БД.
