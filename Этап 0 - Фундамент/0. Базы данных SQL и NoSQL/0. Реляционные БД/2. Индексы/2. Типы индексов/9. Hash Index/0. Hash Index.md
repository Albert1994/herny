# 9. Hash Index

**Раздел:** [[../0. Типы индексов|Типы индексов]]  
**Контекст:** [[../../0. Индексы|Индексы]]

---

## Коротко

**Hash index** ускоряет поиск по точному равенству.

```text
hash(key) -> bucket -> row pointers
```

Он не сохраняет порядок, поэтому не помогает для диапазонов и сортировки.

---

## Для чего нужно

Hash index подходит для запросов:

```sql
WHERE email = 'alice@example.com'
WHERE token = 'abc123'
```

Но не подходит для:

```sql
WHERE created_at BETWEEN ...
ORDER BY created_at
```

---

## PostgreSQL пример

```sql
CREATE INDEX idx_users_email_hash
ON users USING HASH (email);
```

На практике в PostgreSQL чаще используют B-tree даже для `=`, потому что B-tree универсальнее.

---

## Как работает вставка

1. БД считает hash от ключа.
2. Выбирает bucket.
3. Добавляет туда ссылку на строку.
4. При коллизиях хранит несколько значений в bucket.

---

## Как работает получение

```sql
SELECT * FROM users WHERE email = 'alice@example.com';
```

1. Считаем hash от email.
2. Идём в bucket.
3. Проверяем значения внутри bucket, потому что возможны коллизии.
4. Возвращаем подходящие строки.

---

## Ограничения

- Только равенство.
- Нет range scan.
- Нет сортировки.
- Может проиграть B-tree из-за универсальности последнего.

---

## Как объяснить на интервью

Hash index превращает ключ в hash и быстро находит bucket. Он хорош для точного равенства, но не поддерживает порядок, диапазоны и сортировку. Поэтому в SQL-БД B-tree часто остаётся дефолтным выбором даже для `=`.
