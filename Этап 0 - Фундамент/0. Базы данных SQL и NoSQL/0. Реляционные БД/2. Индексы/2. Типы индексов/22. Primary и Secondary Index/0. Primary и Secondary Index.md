# Primary и Secondary Index

**Раздел:** [[../0. Типы индексов|Типы индексов]]  
**Контекст:** [[../../0. Индексы|Индексы]]

---

## Коротко

**Primary index** — основной индекс, через который storage engine организует доступ к строке или записи.

**Secondary index** — дополнительный индекс по другим колонкам/ключам.

Это понятие особенно важно в storage engines: MySQL InnoDB, Cassandra, LSM-хранилищах, key-value engines.

---

## Primary index vs Primary Key

Не путай:

- **Primary Key** — логическое ограничение в SQL: уникальный `NOT NULL` идентификатор.
- **Primary index** — физическая или storage-engine структура доступа.

В MySQL InnoDB они часто совпадают: primary key становится clustered primary index.

В других системах primary index может означать основной путь доступа к партиции/таблице.

---

## Для чего нужно

Разделение primary/secondary важно, потому что secondary index часто не хранит всю строку.

Он хранит:

```text
secondary_key -> primary_key / row_id
```

После нахождения secondary key storage engine делает дополнительный lookup в primary index.

---

## Пример InnoDB

```sql
CREATE TABLE orders (
  id bigint PRIMARY KEY,
  user_id bigint NOT NULL,
  created_at timestamp NOT NULL
);

CREATE INDEX idx_orders_user_id ON orders(user_id);
```

- `PRIMARY KEY(id)` — primary clustered index.
- `idx_orders_user_id` — secondary index.

---

## Как работает получение по primary index

```sql
SELECT * FROM orders WHERE id = 100;
```

1. Идём в primary index по `id`.
2. В листе clustered index уже лежит строка.
3. Возвращаем результат.

---

## Как работает получение по secondary index

```sql
SELECT * FROM orders WHERE user_id = 42;
```

1. Идём в secondary index `idx_orders_user_id`.
2. Получаем список primary key подходящих строк.
3. По каждому primary key идём в primary index.
4. Возвращаем строки.

Это может быть дороже, чем кажется, особенно если найдено много строк.

---

## Cassandra / LSM контекст

В Cassandra primary key определяет:

- partition key — на какой узел и в какую партицию попадут данные;
- clustering columns — порядок внутри партиции.

Secondary indexes в Cassandra имеют ограничения и не работают как универсальные B-tree индексы в PostgreSQL.

---

## Практический вывод

При проектировании таблицы важно понимать:

- какой путь доступа основной;
- какие запросы требуют secondary index;
- сколько дополнительных lookup создаёт secondary index;
- насколько широкий primary key, потому что он может храниться в secondary indexes.

---

## Как объяснить на интервью

Primary index — основной путь доступа storage engine к записи. Secondary index — дополнительная структура по другим ключам, которая часто хранит ссылку на primary key. Поэтому запрос по secondary index может требовать второго lookup в primary/clustered index.
