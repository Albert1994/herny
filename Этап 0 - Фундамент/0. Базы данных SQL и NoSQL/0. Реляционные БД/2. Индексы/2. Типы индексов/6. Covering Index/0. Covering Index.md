# 6. Covering Index

**Раздел:** [[../0. Типы индексов|Типы индексов]]  
**Контекст:** [[../../0. Индексы|Индексы]]

---

## Коротко

**Covering index** — индекс, в котором есть все данные, нужные запросу.

Если БД может ответить только по индексу, не читая таблицу, это называется **Index Only Scan**.

---

## Для чего нужно

Чтение таблицы после индекса может быть дорогим. Covering index уменьшает I/O.

Пример:

```sql
SELECT total_price, status
FROM orders
WHERE user_id = 42;
```

Индекс:

```sql
CREATE INDEX idx_orders_cover
ON orders(user_id)
INCLUDE (total_price, status);
```

---

## PostgreSQL INCLUDE

В PostgreSQL:

```sql
CREATE INDEX idx_orders_user_include
ON orders(user_id)
INCLUDE (total_price, status);
```

`user_id` участвует в поиске и сортировке. `total_price`, `status` просто лежат в листьях индекса, чтобы покрыть SELECT.

---

## Как работает получение

1. БД находит записи в индексе по `user_id`.
2. Видит, что `total_price` и `status` уже есть в индексе.
3. Не идёт в таблицу.

В плане можно увидеть:

```text
Index Only Scan using idx_orders_user_include
```

---

## Важный нюанс PostgreSQL

PostgreSQL использует visibility map. Даже если индекс покрывает запрос, иногда нужно проверить таблицу, если страница не помечена как all-visible.

После `VACUUM` index-only scan обычно становится эффективнее.

---

## Цена

Covering index шире обычного:

- занимает больше места;
- дороже обновляется;
- может ухудшить cache locality.

Не нужно добавлять в `INCLUDE` всё подряд.

---

## Как объяснить на интервью

Covering index содержит все колонки, необходимые запросу. Это позволяет выполнить index-only scan и не читать таблицу. В PostgreSQL дополнительные колонки можно добавить через `INCLUDE`.
