# 2. Foreign Key Index

**Раздел:** [[../0. Типы индексов|Типы индексов]]  
**Контекст:** [[../../0. Индексы|Индексы]]

---

## Коротко

**Foreign Key Index** — индекс на колонку, которая ссылается на другую таблицу.

Сам foreign key — это constraint. Индекс на foreign key — отдельная вещь.

В PostgreSQL индекс на FK **не создаётся автоматически**. В MySQL InnoDB обычно требуется/создаётся индекс для FK.

---

## Для чего нужно

Индекс на FK нужен для:

- быстрых JOIN;
- быстрых проверок ссылочной целостности;
- быстрых `DELETE` / `UPDATE` родительской строки;
- избежания долгих блокировок.

---

## Пример

```sql
CREATE TABLE users (
  id bigint PRIMARY KEY
);

CREATE TABLE orders (
  id bigint PRIMARY KEY,
  user_id bigint NOT NULL REFERENCES users(id)
);

CREATE INDEX idx_orders_user_id ON orders(user_id);
```

Запрос:

```sql
SELECT *
FROM orders
WHERE user_id = 42;
```

или JOIN:

```sql
SELECT u.id, o.id
FROM users u
JOIN orders o ON o.user_id = u.id
WHERE u.id = 42;
```

---

## Как работает получение

Если индекс есть, БД быстро находит все заказы пользователя по `orders.user_id`.

Если индекса нет, БД может сканировать всю таблицу `orders`.

---

## Как работает удаление родителя

```sql
DELETE FROM users WHERE id = 42;
```

БД должна проверить, есть ли заказы у пользователя.

Без индекса на `orders(user_id)` она может пройти всю таблицу заказов.

С индексом — быстро находит связанные строки.

---

## Практическое правило

Почти всегда индексируй foreign key, если:

- по нему есть JOIN;
- по нему есть фильтрация;
- родительские строки удаляются или обновляются;
- таблица-ребёнок большая.

---

## Как объяснить на интервью

Foreign key сам по себе гарантирует ссылочную целостность, но не всегда создаёт индекс. Индекс на FK нужен, чтобы быстро находить дочерние строки при JOIN, DELETE/UPDATE родителя и проверке constraint.
