# 4. Composite Index

**Раздел:** [[../0. Типы индексов|Типы индексов]]  
**Контекст:** [[../../0. Индексы|Индексы]]

---

## Коротко

**Composite index** — индекс по нескольким колонкам.

```sql
CREATE INDEX idx_orders_user_created
ON orders(user_id, created_at);
```

Он полезен, когда запросы фильтруют или сортируют по комбинации колонок.

---

## Главный принцип: leftmost prefix

Индекс `(a, b, c)` хорошо работает для:

```sql
WHERE a = ?
WHERE a = ? AND b = ?
WHERE a = ? AND b = ? AND c = ?
```

Но обычно не работает как полноценный индекс для:

```sql
WHERE b = ?
WHERE c = ?
```

потому что дерево отсортировано сначала по `a`, потом внутри `a` по `b`, потом по `c`.

---

## Как выбирать порядок колонок

Типичный порядок:

1. Колонки с равенством: `user_id = ?`.
2. Потом колонка с диапазоном: `created_at >= ?`.
3. Потом колонки для сортировки/covering, если нужно.

Пример:

```sql
WHERE user_id = 42
  AND created_at >= '2026-01-01'
ORDER BY created_at
```

Хороший индекс:

```sql
CREATE INDEX idx_orders_user_created
ON orders(user_id, created_at);
```

---

## Как работает получение

Для индекса `(user_id, created_at)` дерево хранит ключи примерно так:

```text
(1, Jan 1)
(1, Jan 2)
(2, Jan 1)
(2, Jan 2)
```

Запрос `user_id = 1 AND created_at >= Jan 2` быстро находит область, где `user_id = 1`, и внутри неё делает range scan по дате.

---

## Как работает вставка

При вставке строки БД собирает composite key:

```text
(user_id, created_at)
```

и вставляет его в B+ Tree в отсортированное место. Чем шире ключ, тем больше индекс и дороже запись.

---

## Частая ошибка

Создать индексы:

```sql
CREATE INDEX ON orders(user_id);
CREATE INDEX ON orders(user_id, created_at);
```

Второй индекс часто уже покрывает первый для запросов по `user_id`. Первый может быть лишним.

---

## Как объяснить на интервью

Composite index хранит несколько колонок в одном упорядоченном ключе. Он работает по правилу leftmost prefix: индекс `(a,b,c)` полезен для условий, начинающихся с `a`, но не для `b` без `a`.
