# 10. GIN Index

**Раздел:** [[../0. Типы индексов|Типы индексов]]  
**Контекст:** [[../../0. Индексы|Индексы]]

---

## Коротко

**GIN** — Generalized Inverted Index.

Он индексирует не одно значение строки, а множество элементов внутри строки.

Пример: документ содержит слова, JSONB содержит ключи, массив содержит элементы.

---

## Для чего нужно

GIN нужен для:

- full-text search;
- JSONB;
- массивов;
- tags;
- trigram search;
- случаев, где одна строка раскладывается на много ключей.

---

## Как выглядит

Документы:

```text
doc1: "database index"
doc2: "database architecture"
```

GIN хранит:

```text
"database" -> [doc1, doc2]
"index"    -> [doc1]
"architecture" -> [doc2]
```

---

## JSONB пример

```sql
CREATE INDEX idx_events_payload_gin
ON events USING GIN (payload);
```

Запрос:

```sql
SELECT * FROM events
WHERE payload @> '{"type": "click"}';
```

---

## Array пример

```sql
CREATE INDEX idx_posts_tags_gin
ON posts USING GIN (tags);

SELECT * FROM posts
WHERE tags @> ARRAY['postgres'];
```

---

## Как работает вставка

При вставке строки GIN разбирает значение на элементы:

```text
['postgres', 'index', 'sql']
```

и для каждого элемента обновляет posting list.

Это может быть дороже, чем вставка в B-tree, потому что одна строка даёт много индексных записей.

---

## Как работает получение

Для условия:

```sql
tags @> ARRAY['postgres', 'sql']
```

GIN:

1. находит posting list для `postgres`;
2. находит posting list для `sql`;
3. пересекает списки;
4. возвращает кандидатов;
5. БД может перепроверить условие на таблице.

---

## Ограничения

- Дороже на запись.
- Может требовать настройки pending list / vacuum.
- Не заменяет B-tree для обычного range/order access.

---

## Как объяснить на интервью

GIN — это инвертированный индекс для составных значений: текста, JSONB, массивов. Он хранит элемент → список строк. Поэтому хорошо работает для поиска «какие строки содержат этот элемент», но дороже обновляется.
