# 7. Functional Expression Index

**Раздел:** [[../0. Типы индексов|Типы индексов]]  
**Контекст:** [[../../0. Индексы|Индексы]]

---

## Коротко

**Functional index** или **expression index** — индекс не по колонке, а по выражению.

```sql
CREATE INDEX idx_users_lower_email
ON users (lower(email));
```

Он нужен, когда в `WHERE` используется функция или выражение.

---

## Для чего нужно

Обычный индекс:

```sql
CREATE INDEX idx_users_email ON users(email);
```

может не помочь для:

```sql
WHERE lower(email) = 'alice@example.com'
```

Потому что запрос ищет не `email`, а результат функции `lower(email)`.

---

## Где используется

- case-insensitive email/usernames;
- дата без времени: `date(created_at)`;
- JSONB expression;
- нормализованный телефон;
- вычисляемые бизнес-ключи.

---

## Пример

```sql
CREATE INDEX idx_users_email_ci
ON users (lower(email));

SELECT *
FROM users
WHERE lower(email) = lower('Alice@Example.com');
```

---

## Как работает вставка

При вставке:

```sql
INSERT INTO users(email) VALUES ('Alice@Example.com');
```

БД вычисляет:

```text
lower(email) = 'alice@example.com'
```

и кладёт результат в индекс.

При обновлении `email` выражение пересчитывается.

---

## Как работает получение

Запрос должен совпадать с выражением индекса или быть логически эквивалентен.

Индекс по `lower(email)` не равен индексу по `email`.

---

## Важные ограничения

- Функция должна быть immutable/stable в зависимости от БД и требований.
- Выражение должно совпадать с запросом.
- Чем тяжелее функция, тем дороже вставка и обновление.

---

## Как объяснить на интервью

Expression index хранит результат выражения, а не исходное значение колонки. Он нужен, когда запрос фильтрует по функции вроде `lower(email)` или `date(created_at)`. Без такого индекса БД часто не сможет использовать обычный индекс по колонке.
