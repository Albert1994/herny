# Индексы

**Раздел:** [[0. Реляционные БД|Реляционные БД (SQL)]]

---

## Главная мысль

Индекс — структура данных или метаданных, которая ускоряет доступ к строкам, но занимает место и замедляет запись. Индекс выбирают не «на колонку», а под конкретный паттерн запроса: `WHERE`, `JOIN`, `ORDER BY`, range scan, full-text, spatial, vector search.

Хороший архитектор должен понимать два уровня:

1. **Тип индекса с точки зрения БД** — `B-tree`, `GIN`, `GiST`, `BRIN`, `partial`, `covering`, `unique`, `vector`.
2. **Структура под капотом** — B+ Tree, LSM Tree, R-Tree, Trie, Skip List, HNSW и т.д.

## Базовые правила выбора

1. **Индексируй запрос, а не колонку.** Смотри на реальные `WHERE`, `JOIN`, `ORDER BY`, `GROUP BY`.
2. **Проверяй через `EXPLAIN ANALYZE`.** Не угадывай, использует ли БД индекс.
3. **Помни цену записи.** Каждый `INSERT`, `UPDATE`, `DELETE` должен обновлять индексы.
4. **Следи за селективностью.** Индекс полезен, когда отсекает значительную часть строк.
5. **Порядок колонок в composite index критичен.** Работает leftmost prefix.
6. **Не дублируй индексы.** `(user_id, created_at)` часто покрывает отдельный `(user_id)`.
7. **Для специальных данных нужны специальные индексы.** JSONB ≠ B-tree, гео ≠ B-tree, embeddings ≠ B-tree.

## Практический цикл работы с индексом

```sql
EXPLAIN ANALYZE
SELECT *
FROM orders
WHERE user_id = 42
  AND created_at >= '2026-01-01'
ORDER BY created_at;
```

Если видишь `Seq Scan` на большой таблице — вероятно, нужен индекс.

```sql
CREATE INDEX idx_orders_user_created
ON orders(user_id, created_at);
```

Потом снова:

```sql
EXPLAIN ANALYZE ...
```

Цель — увидеть `Index Scan`, `Index Only Scan`, `Bitmap Index Scan` или другой подходящий план.

## Антипаттерны

- Индекс на каждую колонку «на всякий случай».
- Индекс на колонку с низкой селективностью в OLTP без понимания запроса.
- Несколько почти одинаковых индексов.
- Функция в `WHERE`, но индекс создан на исходную колонку.
- Индекс создан, но запрос не совпадает с его operator class/collation.
- Индекс ускорил один запрос, но убил скорость записи.
- Не проверять реальный план через `EXPLAIN ANALYZE`.
