# Нормализация и денормализация

**Раздел:** [[0. Реляционные БД|Реляционные БД (SQL)]]

---

## Сокращения и термины

| Аббревиатура | Расшифровка | Суть |
|--------------|-------------|------|
| **NF** | Normal Form — нормальная форма | Правило структуры таблицы, убирающее определённый тип избыточности |
| **1NF** | First Normal Form — первая нормальная форма | Атомарные значения, нет повторяющихся групп колонок |
| **2NF** | Second Normal Form — вторая нормальная форма | 1NF + нет частичных зависимостей от составного ключа |
| **3NF** | Third Normal Form — третья нормальная форма | 2NF + нет транзитивных зависимостей |
| **BCNF** | Boyce-Codd Normal Form | 3NF + каждая нетривиальная FD определяется суперключом |
| **4NF** | Fourth Normal Form — четвёртая нормальная форма | BCNF + нет независимых многозначимых зависимостей |
| **5NF** | Fifth Normal Form / **PJNF** (Project-Join NF) | 4NF + нет зависимостей соединения, не выводимых из ключей |
| **6NF** | Sixth Normal Form | Каждый атрибут — отдельная таблица; для temporal DB |
| **FD** | Functional Dependency — функциональная зависимость | A → B: зная A, однозначно определяется B |
| **MVD** | Multivalued Dependency — многозначимая зависимость | A →→ B: у одного A — множество значений B, не зависящих от других атрибутов |
| **JD** | Join Dependency — зависимость соединения | Таблица восстанавливается только через join нескольких проекций |
| **PK** | Primary Key — первичный ключ | Уникальный идентификатор строки |
| **FK** | Foreign Key — внешний ключ | Ссылка на PK другой таблицы |
| **CK** | Candidate Key — потенциальный ключ | Минимальный набор атрибутов, однозначно определяющий строку |
| **SK** | Superkey — суперключ | Набор атрибутов, однозначно определяющий строку (не обязательно минимальный) |
| **OLTP** | Online Transaction Processing | Операционные системы с частыми транзакциями (заказы, платежи) |
| **CTE** | Common Table Expression | Именованный подзапрос в SQL (`WITH ... AS`) |

**Детерминант** — левая часть FD (A в A → B).  
**Тривиальная FD** — B входит в A; **нетривиальная** — B не входит в A.

---

## Главная мысль

Нормализация убирает дубли и аномалии обновления; денормализация — осознанный возврат дублей ради скорости чтения. На практике чаще всего достаточно **3NF или BCNF**; 4NF и 5NF нужны при специфических зависимостях.

---

## Зачем нормализовать

Одна правда в одном месте — обновил значение в одной строке, и везде актуально. Без нормализации возникают **аномалии**:

| Аномалия | Что происходит |
|----------|----------------|
| **Insert** | Нельзя добавить данные без других обязательных полей (нельзя завести продукт без заказа) |
| **Update** | Меняешь значение в одной строке — в других с тем же смыслом остаётся старое |
| **Delete** | Удаляешь запись — теряешь связанные данные, которые нужны сами по себе |

---

## Функциональная зависимость (FD)

Основа всех нормальных форм: **A → B** означает «зная A, однозначно определяется B».

Примеры:
- `product_id → product_name` — имя товара зависит только от id
- `order_id → user_id, order_date` — атрибуты заказа зависят от id заказа
- `(order_id, product_id) → quantity, price` — в строке заказа количество зависит от пары ключей

**Составной ключ** — ключ из нескольких колонок.  
**Частичная зависимость** — атрибут зависит только от *части* составного ключа, а не от всего.  
**Транзитивная зависимость** — A → B → C, где B не является ключом (C «тянется» через неключевое поле).

## Нормальные формы — обзор

| Форма | Требование | Что убирает |
|-------|------------|-------------|
| **1NF** | Атомарные значения, нет повторяющихся групп | Списки и группы в одной ячейке |
| **2NF** | 1NF + нет частичных FD от составного ключа | Зависимость атрибута от части ключа |
| **3NF** | 2NF + нет транзитивных FD | Зависимость через неключевое поле |
| **BCNF** | Каждая нетривиальная FD: детерминант = суперключ | Оставшиеся аномалии, когда детерминант не ключ |
| **4NF** | BCNF + нет независимых MVD | Два независимых «множества» в одной таблице |
| **5NF** | 4NF + нет нетривиальных JD | Тройные и выше связи, не сводимые к парам |
| **6NF** | Каждый атрибут — отдельная таблица | Всё, кроме ключа и одного значения; для temporal |

## Нормальные формы — подробно

### 1NF (First Normal Form)

**Определение:** каждая ячейка содержит одно атомарное значение; нет повторяющихся групп колонок (`phone1`, `phone2`) и составных списков в одном поле.

**Нарушение:** данные «сплющены» — несколько значений в одной ячейке или дублирующиеся колонки для однотипных данных.

**Как исправить:** разбить списки на отдельные строки (отношение «один-ко-многим») или вынести в связанную таблицу.

**Схема:**
```
-- Нарушение 1NF
contacts(id, name, phones)          -- phones = "79001112233, 79004445566"

-- Исправление
contacts(id, name)
contact_phones(contact_id, phone)   -- по одному телефону на строку
```

---

### 2NF (Second Normal Form)

**Определение:** таблица в 1NF, и каждый неключевой атрибут зависит от **всего** составного первичного ключа, а не от его части.

**Нарушение:** составной PK `(A, B)`, но есть FD `A → X` или `B → Y` — атрибут зависит только от части ключа.

**Как исправить:** вынести атрибуты, зависящие от части ключа, в отдельную таблицу с PK = эта часть.

**Схема:**
```
-- Нарушение 2NF
-- PK: (order_id, product_id)
order_items(order_id, product_id, product_name, quantity)
-- product_name зависит только от product_id, не от всей пары

-- Исправление
products(product_id, product_name)
order_items(order_id, product_id, quantity)
```

---

### 3NF (Third Normal Form)

**Определение:** таблица в 2NF, и нет транзитивных зависимостей: неключевой атрибут не должен зависеть от другого неключевого атрибута.

**Нарушение:** `PK → A → B` — B зависит от A, а A не является ключом.

**Как исправить:** вынести промежуточную сущность (A) в отдельную таблицу; в исходной оставить FK.

**Схема:**
```
-- Нарушение 3NF
users(id, name, city_id, city_name)
-- city_name зависит от city_id, а city_id — не ключ users

-- Исправление
users(id, name, city_id)
cities(id, name)
```

**1NF → 3NF** покрывает ~90% реальных OLTP-схем.

---

### BCNF (Boyce-Codd Normal Form)

**Определение:** для каждой нетривиальной FD **A → B** множество A должно быть **суперключом** (уникально определять строку).

**Нарушение:** детерминант (левая часть FD) не является суперключом. Таблица может быть в 3NF, но не в BCNF — когда детерминант неключевой, но при этом «почти ключ» (все его атрибуты входят в неключевые).

**Как исправить:** разбить таблицу так, чтобы каждая FD определялась ключом.

**Схема:**
```
-- Нарушение BCNF
course_schedules(teacher, subject, room)
-- FD: teacher → subject (преподаватель ведёт ровно один предмет)
-- teacher — не суперключ → при смене предмета нужно обновлять все строки

-- Исправление
teachers(teacher, subject)
schedules(teacher, room)
```

**3NF vs BCNF:** каждая BCNF-таблица — 3NF, но не наоборот. BCNF строже и убирает оставшиеся аномалии update/delete.

---

### 4NF (Fourth Normal Form)

**Определение:** таблица в BCNF, и нет **независимых многозначимых зависимостей** (MVD). Запись **A →→ B** означает: у одного A может быть множество B, и это множество не зависит от других атрибутов строки.

**Нарушение:** в одной таблице два независимых «множества» — например, языки и виды спорта студента. Декартово произведение языков × спортов раздувает таблицу без добавления информации.

**Как исправить:** разнести независимые MVD в отдельные таблицы с общим ключом A.

**Схема:**
```
-- Нарушение 4NF
student_skills(student, language, sport)
-- У студента N языков и M видов спорта, но язык не определяет спорт

-- Исправление
student_languages(student, language)
student_sports(student, sport)
```

---

### 5NF (Fifth Normal Form / PJNF)

**Определение:** таблица в 4NF, и нет **нетривиальных зависимостей соединения** (JD). Если связь осмысленна только для **тройки** (или более) атрибутов, а разбиение на парные таблицы даёт «лишние» строки при обратном JOIN — нужна 5NF.

**Нарушение:** трёхсторонняя связь «поставщик S поставляет деталь P для проекта J», которую нельзя корректно восстановить из пар `(S,P)`, `(P,J)`, `(S,J)` без фиктивных комбинаций.

**Как исправить:** оставить одну таблицу с тройным ключом или разбить на проекции, из которых JOIN восстанавливает ровно исходные данные.

**Схема:**
```
-- Связь тройка (supplier, part, project) осмысленна только вместе
supplies(supplier, part, project)

-- Парные проекции sp(supplier, part), pj(part, project), sj(supplier, project)
-- при JOIN дают лишние строки → нужна исходная тройная таблица (5NF)
```

**На практике 5NF встречается редко** — в учебниках и узких доменах (снабжение, сложные many-to-many-to-many).

---

### 6NF (Sixth Normal Form)

**Определение:** каждый неключевой атрибут вынесен в отдельную таблицу вида `(ключ, атрибут, значение)` — иногда с добавлением времени `(ключ, время, атрибут, значение)`.

**Зачем:** **temporal databases** — хранение истории изменений каждого поля независимо. Позволяет отвечать на вопрос «каким было значение X в момент T».

**На практике:** в обычном проектировании не нужна; overhead на JOIN слишком велик для типичных приложений.

**Схема:**
```
-- Вместо employees(id, name, salary, department)
employee_name(id, valid_from, name)
employee_salary(id, valid_from, salary)
employee_department(id, valid_from, department)
```

---

## До какой формы нормализовать?

| Ситуация | Рекомендация |
|----------|--------------|
| OLTP, бизнес-приложения | **3NF** или **BCNF** |
| Сомнения в FD | Проверить на BCNF |
| Независимые списки в одной таблице | Проверить на **4NF** |
| Сложные ternary-связи | Иногда **5NF** |
| Аналитика, отчёты | Часто **денормализация** поверх нормализованного OLTP |

**Правило:** нормализуй OLTP-схему до 3NF/BCNF, затем денормализуй **точечно** под конкретные read-запросы.

## Денормализация

Намеренное дублирование данных ради скорости чтения.

| Подход | Плюсы | Минусы |
|--------|-------|--------|
| Нормализация | Целостность, меньше места, проще обновлять | Больше JOIN, медленнее чтение |
| Денормализация | Быстрое чтение, меньше JOIN | Дубли, сложнее обновлять, риск рассинхрона |

### Техники денормализации

| Техника | Пример | Как синхронизировать |
|---------|--------|----------------------|
| **Дублирование колонки** | `orders.total_price` из `order_items` | Триггер или код при изменении items |
| **Сводная таблица** | `user_order_stats(user_id, order_count, total_spent)` | Периодический пересчёт / event-driven |
| **Материализованное представление** | `CREATE MATERIALIZED VIEW` в PostgreSQL | `REFRESH` по расписанию или инкрементально |
| **Вложенный JSON** | `order.items[]` в одной строке | Для read-heavy, редких обновлений |
| **Вычисляемая колонка** | `GENERATED ALWAYS AS ... STORED` | БД сама поддерживает актуальность |

**Когда денормализовать:** отчёты, дашборды, read-heavy сценарии, где JOIN на миллионах строк — узкое место.

### Риски денормализации

- **Рассинхрон** — `total_price` в orders не совпадает с суммой items
- **Сложность записи** — обновлять нужно несколько мест
- **Больше места** — дубли занимают диск

**Митигация:** единая точка обновления в коде, триггеры, eventual consistency для сводок, reconciliation job.

## Примеры по таблицам

Один сквозной домен — **интернет-магазин** — с данными до и после нормализации.

### 1NF — нарушение и исправление

**До (нарушение):** несколько телефонов в одной ячейке.

| id | name | phones |
|----|------|--------|
| 1 | Alice | 79001112233, 79004445566 |
| 2 | Bob | 79007778899 |

**Проблема:** нельзя искать по одному телефону, нельзя добавить телефон без парсинга строки.

**После (1NF):**

`users`

| id | name |
|----|------|
| 1 | Alice |
| 2 | Bob |

`user_phones`

| user_id | phone |
|---------|-------|
| 1 | 79001112233 |
| 1 | 79004445566 |
| 2 | 79007778899 |

---

### 2NF — нарушение и исправление

**До (нарушение):** PK = `(order_id, product_id)`, но `product_name` зависит только от `product_id`.

`order_items`

| order_id | product_id | product_name | quantity |
|----------|------------|--------------|----------|
| 101 | 5 | Laptop | 1 |
| 101 | 8 | Mouse | 2 |
| 102 | 5 | Laptop | 1 |

**Проблема:** `product_name` дублируется; переименование Laptop → Notebook требует обновления всех строк.

**После (2NF):**

`products`

| product_id | product_name |
|------------|--------------|
| 5 | Laptop |
| 8 | Mouse |

`order_items`

| order_id | product_id | quantity |
|----------|------------|----------|
| 101 | 5 | 1 |
| 101 | 8 | 2 |
| 102 | 5 | 1 |

---

### 3NF — нарушение и исправление

**До (нарушение):** `city_name` зависит от `city_id`, а не от `user_id`.

`users`

| id | name | city_id | city_name |
|----|------|---------|-----------|
| 1 | Alice | 77 | Moscow |
| 2 | Bob | 77 | Moscow |
| 3 | Carol | 78 | SPb |

**Проблема:** Moscow дублируется; переименование города — update во всех строках пользователей.

**После (3NF):**

`cities`

| id | name |
|----|------|
| 77 | Moscow |
| 78 | SPb |

`users`

| id | name | city_id |
|----|------|---------|
| 1 | Alice | 77 |
| 2 | Bob | 77 |
| 3 | Carol | 78 |

---

### BCNF — нарушение и исправление

**До (нарушение):** правило бизнеса — каждый преподаватель ведёт ровно один предмет. FD: `teacher → subject`.

`course_schedules`

| teacher | subject | room |
|---------|---------|------|
| Ivanov | Math | A101 |
| Ivanov | Math | B202 |
| Petrov | Physics | A101 |

**Проблема:** Ivanov сменил предмет с Math на Physics — нужно обновить все его строки; при insert новой аудитории для Ivanov+Math дублируется subject.

**После (BCNF):**

`teachers`

| teacher | subject |
|---------|---------|
| Ivanov | Math |
| Petrov | Physics |

`schedules`

| teacher | room |
|---------|------|
| Ivanov | A101 |
| Ivanov | B202 |
| Petrov | A101 |

---

### 4NF — нарушение и исправление

**До (нарушение):** языки и спорты студента независимы, но записаны в одной таблице → декартово произведение.

`student_skills`

| student | language | sport |
|---------|----------|-------|
| Alice | English | Tennis |
| Alice | English | Swimming |
| Alice | German | Tennis |
| Alice | German | Swimming |

**Проблема:** 4 строки вместо 2+2; добавить Spanish → нужно 2 новые строки (по одной на каждый спорт).

**После (4NF):**

`student_languages`

| student | language |
|---------|----------|
| Alice | English |
| Alice | German |

`student_sports`

| student | sport |
|---------|-------|
| Alice | Tennis |
| Alice | Swimming |

---

### 5NF — нарушение и исправление

**До:** поставщик S поставляет деталь P **именно для** проекта J (тройка осмысленна).

`supplies`

| supplier | part | project |
|----------|------|---------|
| Acme | Bolt | Bridge |
| Acme | Nut | Bridge |
| Beta | Bolt | Tower |

**Проблема при разбиении на пары:** если сделать `sp(supplier, part)` и `pj(part, project)`, JOIN даст `(Acme, Bolt, Tower)` — фиктивная строка, Acme не поставляет Bolt для Tower.

**После (5NF):** тройная таблица `supplies` остаётся единственным источником правды; парные проекции используются только для чтения через view, не как независимое хранение.

---

### 6NF — пример (temporal)

**До:** обычная таблица без истории.

`employees`

| id | name | salary | department |
|----|------|--------|------------|
| 1 | Alice | 100000 | Engineering |

**После (6NF + время):** каждый атрибут — своя таблица с периодом действия.

`employee_salary`

| employee_id | valid_from | valid_to | salary |
|-------------|------------|----------|--------|
| 1 | 2025-01-01 | 2025-06-30 | 100000 |
| 1 | 2025-07-01 | NULL | 120000 |

`employee_department`

| employee_id | valid_from | valid_to | department |
|-------------|------------|----------|------------|
| 1 | 2025-01-01 | NULL | Engineering |

**Запрос:** «Какая была зарплата Alice 2025-03-15?» → `WHERE valid_from <= '2025-03-15' AND (valid_to IS NULL OR valid_to > '2025-03-15')`.

## Чеклист при проектировании

1. Выдели сущности и связи (ER-модель)
2. Приведи к **1NF** — атомарные поля, вынеси повторяющиеся группы
3. Проверь **2NF** — если составной ключ, нет ли частичных зависимостей
4. Проверь **3NF** — нет ли транзитивных зависимостей через неключевые поля
5. При сомнениях — **BCNF**
6. Независимые «списки» в одной строке — **4NF**
7. Для hot read-запросов — точечная **денормализация** или materialized view
