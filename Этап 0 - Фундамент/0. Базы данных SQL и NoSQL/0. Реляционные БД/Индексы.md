# Индексы

**Раздел:** [[0. Реляционные БД|Реляционные БД (SQL)]]

---

## Главная мысль

Индекс — структура данных, которая ускоряет поиск, но замедляет запись и занимает место. Индексируй то, по чему реально фильтруешь и сортируешь в `WHERE` / `JOIN` / `ORDER BY`.

---

## B-tree

Balanced Tree — стандартный индекс в PostgreSQL, MySQL, SQLite.

- Дерево сбалансировано → поиск за O(log n)
- Поддерживает диапазоны: `WHERE age BETWEEN 18 AND 65`, `ORDER BY created_at`
- Листья хранят указатели на строки таблицы (или сами данные в clustered index)

---

## Composite index (составной)

Индекс по нескольким колонкам: `(user_id, created_at)`.

- Работает по принципу **leftmost prefix**: индекс `(a, b, c)` помогает для `WHERE a = ?`, `WHERE a = ? AND b = ?`, но не для `WHERE b = ?` без `a`
- Порядок колонок важен: сначала — самые селективные или те, что в `WHERE` с равенством

---

## Partial index

Индекс только на подмножество строк: `WHERE status = 'active'`.

- Меньше размер, быстрее обновление
- Идеален, когда запросы всегда фильтруют по одному условию

---

## Covering index (index-only scan)

Индекс содержит все колонки, нужные запросу.

- PostgreSQL: `CREATE INDEX ... INCLUDE (column)` или все нужные колонки в индексе
- БД читает только индекс, не лезет в таблицу → меньше I/O

---

## Антипаттерны

- Индекс на каждую колонку
- Индекс на колонку с низкой селективностью (`gender`)
- Индекс, который не используется в плане запроса (проверять через `EXPLAIN`)

---

## Практика

```sql
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 42 AND created_at > '2025-01-01';
-- seq scan → нужен индекс
-- index scan / index only scan → индекс работает
```
