# Column-family DB

**Раздел:** [[NoSQL]]

---

## Главная мысль

В Cassandra сначала пишешь запросы, потом проектируешь таблицы. Одни и те же данные могут лежать в нескольких таблицах под разные query patterns.

---

## Модель

Данные организованы по **partition key** → строки с динамическими колонками. Оптимизированы под запись и чтение по ключу партиции.

**Примеры:** Cassandra, HBase, ScyllaDB.

---

## Когда использовать

- Write-heavy: логи, метрики, time-series, IoT-события
- Данные с естественным partition key: `user_id + date`, `device_id + timestamp`
- Нужна высокая доступность и линейное масштабирование на запись

---

## Плюсы и минусы

| Плюсы | Минусы |
|-------|--------|
| Огромный throughput на запись | Запросы ограничены моделью данных |
| Отказоустойчивость | Eventual consistency по умолчанию |
| Линейное масштабирование | Схема проектируется под запросы, не наоборот |

---

## Query-first design

```sql
-- Запрос: события пользователя за день
SELECT * FROM events_by_user
WHERE user_id = ? AND date = ?;

-- Таблица спроектирована под этот запрос:
-- PARTITION KEY (user_id, date), CLUSTERING KEY (timestamp)
```

Один и тот же event может дублироваться в `events_by_user` и `events_by_device` — это норма.
