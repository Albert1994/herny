# Key-Value DB

**Раздел:** [[NoSQL]]

---

## Главная мысль

KV — самый быстрый слой. Используй как кэш или для данных с простым ключом доступа, не как основное хранилище бизнес-сущностей (кроме DynamoDB под конкретные access patterns).

---

## Модель

`key → value`, без структуры запросов — только get/set по ключу.

**Примеры:** Redis, DynamoDB, Memcached, etcd.

---

## Когда использовать

- Кэш (результат тяжёлого запроса, HTML-фрагмент)
- Сессии пользователей
- Счётчики, rate limiting, leaderboards (`INCR`)
- Очереди, pub/sub (Redis)
- Idempotency keys, distributed locks

---

## Плюсы и минусы

| Плюсы | Минусы |
|-------|--------|
| Экстремально быстрый доступ O(1) | Нет сложных запросов |
| Простая модель | Redis: данные в RAM — ограничен объём |
| TTL из коробки | Нужна стратегия персистентности |

---

## Redis vs DynamoDB

| | Redis | DynamoDB |
|---|-------|----------|
| Хранение | In-memory (+ опционально на диск) | На диске, managed |
| Масштаб | Vertical + Cluster | Horizontal из коробки |
| Типичная роль | Кэш, сессии, locks | Primary store под key-based access |
