# Graph DB

**Раздел:** [[NoSQL]]

---

## Главная мысль

Если главный вопрос — «кто с кем связан и через сколько шагов» — графовая БД эффективнее SQL с рекурсивными CTE.

---

## Модель

Узлы (nodes) и рёбра (edges) со свойствами. Запросы обходят граф.

**Примеры:** Neo4j, Amazon Neptune, ArangoDB.

---

## Когда использовать

- Социальные связи: «друзья друзей», «кто на кого подписан»
- Рекомендации: «люди, купившие X, также купили Y»
- Fraud detection: цепочки подозрительных транзакций
- Оргструктуры, права доступа, зависимости в инфраструктуре

---

## Плюсы и минусы

| Плюсы | Минусы |
|-------|--------|
| Обход связей за O(1) на hop | Не для bulk analytics по всей таблице |
| Выразительный язык (Cypher, Gremlin) | Сложнее горизонтальное масштабирование |
| Естественная модель для связей | Узкая ниша |

---

## Пример запроса (Cypher)

```cypher
// Друзья друзей (2 hops)
MATCH (me:User {id: $id})-[:FRIENDS_WITH*1..2]-(friend)
RETURN DISTINCT friend
```

В SQL это потребовало бы рекурсивный CTE или несколько JOIN — медленнее на глубоких графах.
