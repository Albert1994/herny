# Нормализация и денормализация

**Раздел:** [[Реляционные БД|Реляционные БД (SQL)]]

---

## Главная мысль

Нормализация убирает дубли и аномалии обновления; денормализация — осознанный возврат дублей ради скорости чтения. На практике чаще всего достаточно **3NF или BCNF**; 4NF и 5NF нужны при специфических зависимостях.

---

## Зачем нормализовать

Одна правда в одном месте — обновил значение в одной строке, и везде актуально. Без нормализации возникают **аномалии**:

| Аномалия | Что происходит |
|----------|----------------|
| **Insert** | Нельзя добавить данные без других обязательных полей (нельзя завести продукт без заказа) |
| **Update** | Меняешь значение в одной строке — в других с тем же смыслом остаётся старое |
| **Delete** | Удаляешь запись — теряешь связанные данные, которые нужны сами по себе |

---

## Функциональная зависимость (FD)

Основа всех нормальных форм: **A → B** означает «зная A, однозначно определяется B».

Примеры:
- `product_id → product_name` — имя товара зависит только от id
- `order_id → user_id, order_date` — атрибуты заказа зависят от id заказа
- `(order_id, product_id) → quantity, price` — в строке заказа количество зависит от пары ключей

**Составной ключ** — ключ из нескольких колонок. **Частичная зависимость** — атрибут зависит только от части составного ключа. **Транзитивная зависимость** — A → B → C, где B не является ключом.

---

## Нормальные формы

| Форма | Требование | Пример нарушения |
|-------|------------|------------------|
| **1NF** | Атомарные значения, нет повторяющихся групп | `tags: "js,python,go"` в одной ячейке; несколько телефонов в колонках `phone1`, `phone2` |
| **2NF** | 1NF + нет частичных зависимостей от составного ключа | `(order_id, product_id) → product_name` — имя зависит только от `product_id` |
| **3NF** | 2NF + нет транзитивных зависимостей | `user_id → city_id → city_name` — город зависит от `city_id`, не от пользователя |
| **BCNF** | 3NF + каждая нетривиальная FD определяется **суперключом** | `teacher, subject → room` при правиле «один преподаватель ведёт один предмет в одной аудитории» |
| **4NF** | BCNF + нет **многозначимых зависимостей** (MVD) | `student →→ course` и `student →→ hobby` независимо друг от друга в одной таблице |
| **5NF** | 4NF + нет **зависимостей соединения** (join dependency), которые нельзя вывести из ключей | Данные восстанавливаются только через join трёх+ таблиц, а не пар |

### 1NF → 3NF (основа для 90% проектов)

```
-- Нарушение 3NF
users(id, name, city_id, city_name)

-- Исправление
users(id, name, city_id)
cities(id, name)
```

### BCNF (Boyce-Codd Normal Form)

Строже 3NF. Нарушение бывает, когда **детерминант** (левая часть FD) не является суперключом.

```
-- Нарушение BCNF
course_schedules(teacher, subject, room)
-- FD: teacher → subject  (каждый преподаватель ведёт ровно один предмет)
-- teacher — не суперключ → аномалии при смене предмета у преподавателя

-- Исправление
teachers(teacher, subject)
schedules(teacher, room)
```

**3NF vs BCNF:** каждая BCNF-таблица — 3NF, но не наоборот. BCNF убирает оставшиеся аномалии, когда детерминант — не ключ.

### 4NF (Fourth Normal Form)

Убирает **независимые многозначимые зависимости** (A →→ B | C — зная A, множество B не зависит от C).

```
-- Нарушение 4NF
student_skills(student, language, sport)
-- У студента несколько языков и несколько видов спорта, но язык не определяет спорт

-- Исправление
student_languages(student, language)
student_sports(student, sport)
```

На практике 4NF важна, когда в одной таблице смешаны **два независимых «множества»** атрибутов.

### 5NF (Fifth / PJNF)

Убирает **зависимости соединения** — когда строку нельзя разбить на две таблицы без потери информации, но можно на три и более.

```
-- Пример: поставщики, детали, проекты
-- supplier S поставляет part P для project J
-- Тройка (S, P, J) осмысленна только вместе, но парные проекции теряют контекст

supplies(supplier, part, project)  -- 5NF, если join из парных таблиц даёт «лишние» строки
```

**5NF на практике встречается редко** — в основном в учебных примерах и специфичных доменах.

### 6NF (для справки)

Каждая неключевая колонка — отдельная таблица с ключом и значением. Используется в **temporal databases** (история изменений по времени). В обычном проектировании не нужна.

## До какой формы нормализовать?

| Ситуация | Рекомендация |
|----------|--------------|
| OLTP, бизнес-приложения | **3NF** или **BCNF** |
| Сомнения в FD | Проверить на BCNF |
| Независимые списки в одной таблице | Проверить на **4NF** |
| Сложные ternary-связи | Иногда **5NF** |
| Аналитика, отчёты | Часто **денормализация** поверх нормализованного OLTP |

**Правило:** нормализуй OLTP-схему до 3NF/BCNF, затем денормализуй **точечно** под конкретные read-запросы.

## Денормализация

Намеренное дублирование данных ради скорости чтения.

| Подход | Плюсы | Минусы |
|--------|-------|--------|
| Нормализация | Целостность, меньше места, проще обновлять | Больше JOIN, медленнее чтение |
| Денормализация | Быстрое чтение, меньше JOIN | Дубли, сложнее обновлять, риск рассинхрона |

### Техники денормализации

| Техника | Пример | Как синхронизировать |
|---------|--------|----------------------|
| **Дублирование колонки** | `orders.total_price` из `order_items` | Триггер или код при изменении items |
| **Сводная таблица** | `user_order_stats(user_id, order_count, total_spent)` | Периодический пересчёт / event-driven |
| **Материализованное представление** | `CREATE MATERIALIZED VIEW` в PostgreSQL | `REFRESH` по расписанию или инкрементально |
| **Вложенный JSON** | `order.items[]` в одной строке | Для read-heavy, редких обновлений |
| **Вычисляемая колонка** | `GENERATED ALWAYS AS ... STORED` | БД сама поддерживает актуальность |

**Когда денормализовать:** отчёты, дашборды, read-heavy сценарии, где JOIN на миллионах строк — узкое место. Всегда осознанный trade-off, не «по умолчанию».

### Риски денормализации

- **Рассинхрон** — `total_price` в orders не совпадает с суммой items
- **Сложность записи** — обновлять нужно несколько мест
- **Больше места** — дубли занимают диск

**Митигация:** единая точка обновления в коде, триггеры, eventual consistency для сводок, периодическая сверка (reconciliation job).

## Чеклист при проектировании

1. Выдели сущности и связи (ER-модель)
2. Приведи к **1NF** — атомарные поля, вынеси повторяющиеся группы
3. Проверь **2NF** — если составной ключ, нет ли частичных зависимостей
4. Проверь **3NF** — нет ли транзитивных зависимостей через неключевые поля
5. При сомнениях — **BCNF**
6. Независимые «списки» в одной строке — **4NF**
7. Для hot read-запросов — точечная **денормализация** или materialized view
