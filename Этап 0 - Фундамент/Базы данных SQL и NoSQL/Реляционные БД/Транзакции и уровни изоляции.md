# Транзакции и уровни изоляции

**Раздел:** [[Реляционные БД|Реляционные БД (SQL)]]

---

## Главная мысль

Чем выше изоляция — тем меньше аномалий, но больше блокировок и ниже throughput. Для большинства CRUD достаточно Read Committed; Serializable — когда цена ошибки высока (двойное списание денег, двойное бронирование).

---

## Транзакция

Атомарная единица работы: либо все изменения применяются (**COMMIT**), либо ни одно (**ROLLBACK**).

### ACID

| Буква | Значение | Суть |
|-------|----------|------|
| **A** | Atomicity | Всё или ничего |
| **C** | Consistency | БД переходит из одного валидного состояния в другое (constraints, FK) |
| **I** | Isolation | Параллельные транзакции не мешают друг другу |
| **D** | Durability | После COMMIT данные переживут сбой |

---

## Проблемы без изоляции

- **Dirty read** — читаешь незакоммиченные данные другой транзакции
- **Non-repeatable read** — два одинаковых `SELECT` в одной транзакции дают разный результат
- **Phantom read** — появляются новые строки, подходящие под условие
- **Lost update** — две транзакции перезаписывают одно поле, одна теряется

---

## Уровни изоляции (от слабого к сильному)

| Уровень | Dirty read | Non-repeatable | Phantom | Типичное использование |
|---------|------------|----------------|---------|------------------------|
| Read Uncommitted | ✓ возможен | ✓ | ✓ | Почти не используется |
| **Read Committed** | ✗ | ✓ возможен | ✓ | **Дефолт в PostgreSQL** |
| Repeatable Read | ✗ | ✗ | ✓* | Дефолт в MySQL InnoDB |
| **Serializable** | ✗ | ✗ | ✗ | Финансы, бронирования, строгая корректность |

\* В PostgreSQL Repeatable Read блокирует и phantom reads через snapshot isolation.
