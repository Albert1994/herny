## Главная мысль

В распределённой системе при сетевом разделении (partition) нельзя одновременно гарантировать и строгую согласованность всех узлов, и 100% доступность. Это не «выбор технологии навсегда», а понимание, что система жертвует в момент сбоя сети.

---

## Три свойства

| Буква | Свойство | Суть |
|-------|----------|------|
| **C** | Consistency | Все узлы видят одни и те же данные в один момент (любой read возвращает последний write) |
| **A** | Availability | Каждый запрос (на любой живой узел) получает ответ, даже если другие узлы недоступны |
| **P** | Partition tolerance | Система продолжает работать при разрыве связи между узлами |

---

## Теорема CAP

При **network partition** (P — неизбежно в реальных сетях) остаётся выбор только между **C** и **A**:

```
        Partition произошёл
              │
    ┌─────────┴─────────┐
    ▼                   ▼
 CP (Consistency)   AP (Availability)
 Отклонить запрос    Ответить, но данные
 или ждать синхр.    могут быть устаревшими
```

**CA-системы** (без partition tolerance) существуют только на одном узле — это не распределённая система в полном смысле.

---

## На практике — спектр, не бинарный выбор

- **PostgreSQL** (single node / синхронная репликация) → **CP**: при partition реплика не принимает записи, пока primary не подтвердит
- **Cassandra** → **AP** (настраиваемо): при partition узлы отвечают, согласованность догоняется позже (eventual consistency)
- **Redis Cluster** → при partition возможны split-brain и потеря части данных без правильной настройки

---

## Практический выбор под задачу

| Задача | Выбор | Почему |
|--------|-------|--------|
| Платежи, баланс счёта | **PostgreSQL** (CP) | Потеря или рассинхрон денег недопустимы — лучше ошибка, чем неверный баланс |
| Каталог товаров, профили | **MongoDB** / PostgreSQL | MongoDB — если схема гибкая и eventual consistency допустима |
| Сессии, кэш, rate limit | **Redis** (AP) | Скорость важнее; потеря сессии = пользователь перелогинится |
| Логи, аналитика событий | **Cassandra** (AP) | Write-heavy, потеря единичного события менее критична |
| Рекомендации «связанные товары» | **Neo4j** / PostgreSQL | Граф — если связи сложные; PostgreSQL — если связей мало |

## Вопросы для выбора хранилища

1. Что дороже — временная недоступность или неверные данные?
2. Какой паттерн доступа: по ключу, по связям, по диапазону, агрегации?
3. Нужны ли транзакции между несколькими сущностями?
4. Какой объём записи и чтения (RPS, размер данных)?
