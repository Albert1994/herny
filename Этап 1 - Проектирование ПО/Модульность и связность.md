# Модульность и связность

**Этап:** [[00 Обзор этапа|1 — Проектирование ПО]]  
**Время:** ~1 неделя

---

## Цель

Проектировать систему из loosely coupled, highly cohesive модулей — фундамент для монолита и микросервисов.

---

## Ключевые концепции

### Метрики модульности
- **Afferent coupling (Ca)** — кто зависит от модуля
- **Efferent coupling (Ce)** — от кого зависит модуль
- **Instability** I = Ce / (Ca + Ce)
- **Abstractness** A = abstract types / total types
- **Distance from main sequence** D = |A + I − 1|

### Package principles (Robert Martin)
- **REP** — Release Reuse Equivalency
- **CCP** — Common Closure Principle
- **CRP** — Common Reuse Principle
- **ADP** — Acyclic Dependencies Principle
- **SDP** — Stable Dependencies Principle
- **SAP** — Stable Abstractions Principle

### Практика
- Module boundaries = bounded contexts (часто)
- Public API модуля минимален
- Internal implementation hidden
- Dependency rule: зависимости направлены «внутрь»

---

## Материалы

- Robert Martin — *Clean Architecture* (главы про компоненты)
- Neal Ford et al. — *Building Evolutionary Architectures* (fitness functions)
- Structurizr / dependency-cruiser — визуализация зависимостей

## Практика

1. Построить dependency graph проекта (Mermaid или tool)
2. Найти циклические зависимости и разорвать
3. Выделить модуль с чётким public API

---

## Критерии освоения

- [ ] Читаю dependency graph и вижу проблемные связи
- [ ] Могу объяснить CCP и применить при группировке классов
- [ ] В проекте нет circular dependencies между модулями
