## Зачем этот этап

Микросервисы и cloud-native системы — это распределённые системы с inherent complexity: сбои сети, partial failures, eventual consistency. Архитектор должен проектировать с учётом этого, а не надеяться на «надёжную сеть».

## Проект этапа: Распределённый сервис с observability

### Задача

Развернуть 2–3 сервиса с:
- Message broker (RabbitMQ / Kafka)
- Redis cache
- Prometheus metrics + Grafana dashboard
- Structured logging с correlation ID
- Chaos test: kill one service, document behavior

### Критерии сдачи

- [ ] Dashboard с latency, error rate, throughput
- [ ] Trace одного request через все сервисы
- [ ] Runbook: «что делать при падении Service X»
