# Коммуникация со стейкхолдерами

**Этап:** [[00 Обзор этапа|5 — Практика архитектора]]  
**Время:** ~2 недели

---

## Цель

Объяснять технические решения бизнесу и переводить business needs в architectural requirements — elevator architect skill.

---

## Ключевые концепции

### Stakeholder map
| Stakeholder | Interest | Communication |
|-------------|----------|---------------|
| CTO / VP Eng | Strategy, cost, risk | Executive summary, trade-offs |
| Product Manager | Time-to-market, features | Impact on roadmap |
| Developers | Implementation clarity | C4, ADR, design docs |
| Ops / SRE | Reliability, ops burden | Runbooks, SLO, deployment |
| Security | Compliance, risk | Threat model, audit |
| Business | ROI, competitive advantage | Business case |

### Communication formats
- **Executive summary** — 1 page, no jargon
- **Architecture Decision Record** — for tech audience
- **RFC** — proposal seeking feedback
- **Workshop** — Event Storming, QAW for NFR

### The Software Architect Elevator (Hohpe)
- Travel between «penthouse» (business) and «engine room» (tech)
- Translate, don't dictate
- Build trust before pushing for change

### Presenting trade-offs
```
Option A: Faster delivery, higher tech debt
Option B: Slower, better long-term maintainability
Recommendation: B because [business reason]
```

### Soft skills
- Active listening
- Facilitation over authority
- Influence without formal power
- Saying «no» with alternatives
