## Цель

Планировать эволюцию систем без big-bang rewrite — pragmatic migration paths.

---

## Ключевые концепции

### Migration patterns
- **Strangler Fig** — постепенная замена legacy новым кодом через facade/proxy
- **Branch by Abstraction** — abstraction layer, swap implementation
- **Parallel Run** — old and new run together, compare results
- **Event Interception** — intercept events, route to new system
- **Database migration** — dual write, change data capture (CDC)

### Monolith → Microservices
1. Identify bounded contexts (don't split by layer!)
2. Extract one module at a time
3. Start with leaf services (fewest dependencies)
4. Anti-corruption layer for legacy integration
5. Don't extract until module boundaries are clean in monolith

### Cloud migration
- **Rehost** (lift-and-shift)
- **Replatform** (managed DB, containers)
- **Refactor/Re-architect**
- **Repurchase** (SaaS replacement)
- **Retire** — turn off unused systems

### Risk management
- Feature flags for gradual rollout
- Rollback plan always defined
- Incremental value delivery (not multi-year big bang)
- Measure: error rate, latency during migration

### The Incremental Architect
- Evolutionary architecture > big design upfront
- Fitness functions guard constraints during change
